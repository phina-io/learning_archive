## 1. GitHub 실습: 이슈 발급부터 브랜치 삭제까지의 협업 흐름

GitHub은 단순한 코드 저장소를 넘어 팀원들과 소통하며 코드를 점검하는 강력한 협업 툴입니다. 오늘은 아래의 3가지 핵심 기능을 중심으로 전체 프로세스를 직접 실습했습니다.

### 💡 핵심 협업 개념 정리
* Issue(이슈): 프로젝트에서 해결해야 할 버그, 새로 추가할 기능, 개선 사항 등을 기록하고 논의하는 '작업 게시판'입니다.

* Pull Request(PR / 풀 리퀘스트): 내가 변경한 코드(브랜치)를 메인 코드(`main`)에 합쳐달라고 팀원들에게 '병합 요청'을 보내는 기능입니다.

* Code Review(코드 리뷰): PR을 받은 팀원들이 코드를 안전하게 반영하기 전에 변경 사항을 확인하고 의견을 남기거나 승인(Approve)을 하는 품질 검증 과정입니다.

---


### 🔄 실습 진행 순서 요약
1. 저장소 생성 및 초기화: GitHub 홈페이지 자체에서 원격 저장소를 만들고 `README.md` 파일을 함께 생성했습니다.

2. 셀프 이슈(Issue) 발급: "README 파일의 내용이 부족하다"는 개선 요청 사항으로 스스로 이슈를 발행하여 작업 티켓을 만들었습니다.

3. 작업용 브랜치 생성: 안정적인 `main` 브랜치에 바로 작업하지 않고 문서 수정을 위한 별도의 `docs` 브랜치를 로컬에서 생성하여 작업 공간을 분리했습니다.

4. 수정 및 푸시(Push): `docs` 브랜치에서 README 내용을 업데이트한 후 원격 저장소의 `docs` 브랜치로 코드를 업로드했습니다.

5. 풀 리퀘스트(PR) 및 리뷰: GitHub에서 `main` 브랜치로 합쳐달라는 PR을 보낸 뒤 변경 사항을 검토(Code Review)했습니다.

6. 병합(Merge) 및 삭제: 성공적으로 머지가 완료된 후 역할을 다한 원격 저장소의 `docs` 브랜치를 깔끔하게 삭제하여 저장소를 관리했습니다.


* Issue (이슈): 프로젝트에서 해결해야 할 버그, 새로 추가할 기능, 개선 사항 등을 기록하고 논의하는 '작업 게시판'입니다.

* Pull Request(PR / 풀 리퀘스트): 내가 변경한 코드(브랜치)를 메인 코드(`main`)에 합쳐달라고 팀원들에게 '병합 요청'을 보내는 기능입니다

* Code Review (코드 리뷰): PR을 받은 팀원들이 코드를 안전하게 반영하기 전에 변경 사항을 확인하고 의견을 남기거나 승인(Approve)을 하는 품질 검증 과정입니다.

## 2. 터미널 필수 명령어 정리 💻


### ① 운영체제(OS) 및 터미널 기본 명령어
- Git을 다루기 전, 터미널 환경(CLI)에서 디렉토리를 이동하고 파일을 제어하는 가장 기초적인 명령어입니다.
```bash
# [조회] 현재 위치한 폴더의 파일 및 하위 폴더 목록 출력
$ ls   

# [이동] 상위 폴더(부모 디렉토리)로 한 단계 빠져나감
$ cd .. 

# [생성] 현재 위치에 'HTML'이라는 이름의 새 폴더(디렉토리) 생성
$ mkdir HTML 

# [이동] 'HTML'이라는 이름의 하위 폴더로 진입                          
$ cd HTML                            
                             
# [윈도우 탐색기] 현재 터미널 위치를 윈도우 파일 탐색기 창으로 즉시 열기                        
$ explorer .      

# [VS Code 실행] 현재 폴더를 기준으로 비주얼 스튜디오 코드를 즉시 실행
$ code .
```
>💡 OS별 code . 설정 차이점:
Windows: 설치 시 기본 옵션으로 지정했다면 별도 세팅 없이 터미널에서 즉시 동작합니다.
>
> macOS: VS Code 실행 후 Cmd + Shift + P를 눌러 명렴 팔레트를 열고 Shell Command: Install 'code' command in PATH를 수동으로 설치해 주어야 터미널에서 실행됩니다.

### ② Git 초기 설정 및 저장소 생성
- 컴퓨터에 Git을 처음 설치했거나 새로운 프로젝트(저장소)를 시작할 때 사용하는 명령어입니다.
```bash
# [전역 설정] 커밋 작성자 식별을 위한 사용자 이름 등록
$ git config --global user.name "이름"   

# [전역 설정] 커밋 작성자 식별을 위한 사용자 이메일 주소 등록
$ git config --global user.email "이메일" 

# [확인] 현재 등록된 전역 사용자 이름 확인
$ git config --global user.name   

# [확인] 현재 등록된 전역 사용자 이메일 확인
$ git config --global user.email        

# [저장소 생성] 현재 폴더를 Git이 추적하는 '로컬 저장소'로 초기화 (.git 폴더 생성)
$ git init                              
```

### ③ 로컬 버전 관리(추적 ➡️ 대기 ➡️ 저장)
- 내 컴퓨터 안에서 파일의 변경 사항을 추적하고 버전을 기록(세이브포인트 생성)하는 핵심 흐름입니다.
```bash   
# [상태 확인] 수정된 파일, 스테이징 여부, 커밋 가능 상태를 시각적으로 점검
# (💡 스테이징 전은 🔴붉은색, 스테이징 완료는 🟢초록색 문자로 표시됨)
$ git status                            

# [스테이징] 특정 파일('main.py')만 커밋 대기 공간(Staging Area)에 추가
$ git add main.py     

# [스테이징] 변경되거나 새로 생성된 모든 파일을 한 번에 스테이징 영역에 추가
$ git add .                             

# [커밋] 스테이징 영역의 파일들을 메시지와 함께 하나의 '버전'으로 로컬 저장소에 영구 기록
$ git commit -m "Feat: 기능 추가"       
```

### ④ 커밋 이력 및 세부 내역 조회
- 과거에 쌓아온 커밋 기록을 조회하거나 어떤 부분이 변경되었는지 분석할 때 사용하는 명령어입니다.
```bash  
# [전체 조회] 상세한 전체 커밋 로그(작성자, 날짜, 고유 커밋 해시, 메시지) 출력
$ git log  

# [요약 조회] 커밋 이력을 한 줄씩 아이디와 메시지만 깔끔하게 요약해서 출력
$ git log --oneline                     

 # [상세 확인] 특정 커밋 하나를 지정해서 어떤 파일의 어느 코드가 바뀌었는지 확인
$ git show <커밋해시>     

# [차이점 비교] 작업 디렉토리와 스테이징 영역 간의 코드 차이점 분석
$ git diff      

# [추적] 'main.py' 파일의 각 줄을 누가, 언제, 어떤 커밋에서 수정했는지 행별로 확인
$ git blame main.py                     
```

### ⑤ 브랜치(Branch) 제어 및 관리
- 독립된 작업 공간을 만들고 이동하며, 역할을 다한 공간을 정리하는 명령어입니다.
```bash   
# [조회] 로컬에 존재하는 모든 브랜치 목록 확인 (* 표시가 현재 내가 있는 곳)
$ git branch           

# [생성] 'feature/login' 이라는 이름의 새로운 작업 브랜치 생성
$ git branch feature/login   

# [이동] 작업 공간을 'feature/login' 브랜치로 전환 (과거 checkout 명령어)
$ git switch feature/login              

# [안전 삭제] 병합(Merge)이 완료되어 역할을 다한 로컬 브랜치 안전하게 삭제
$ git branch -d feature/login     

# [강제 삭제] 병합 여부와 관계없이 해당 로컬 브랜치를 강제로 완전 삭제
$ git branch -D feature/login     

# [도움말] 브랜치 제어 관련 Git 공식 매뉴얼/가이드를 터미널에 출력
$ git branch --help                     
 ```

 ### ⑥ 원격 저장소(GitHub) 연동 및 동기화
 내 컴퓨터(로컬)의 이력을 클라우드(원격)에 백업하거나, 팀원들과 소통하며 데이터를 주고받는 명령어입니다
```bash 
# [원격 연결] 로컬 저장소와 외부 GitHub 원격 저장소를 최초로 연결   
$ git remote add origin <원격주소>     

# [주소 확인] 현재 연결되어 있는 원격 저장소(GitHub)의 이름과 URL 주소 목록 출력
$ git remote -v                         

$ git push origin main                  # [업로드] 로컬 저장소의 최신 커밋 이력들을 원격 저장소(`origin`)의 `main` 브랜치로 반영
$ git pull origin main                  # [다운로드] 원격 저장소의 최신 변경 사항을 가져와 현재 나의 로컬 코드와 자동으로 병합
```

## 3. HTML 웹 문서 구성 기초 실습 🌐
HTML(HyperText Markup Language)은 웹 페이지의 '뼈대(구조)'를 만드는 가장 기본적인 마크다운 언어입니다. 
단순히 글자를 적는 것이 아니라 각 자리에 맞는 태그(Tag)를 사용해 컴퓨터에게 문서의 구조를 알려줍니다.

오늘 실습한 코드를 바탕으로 각 태그가 가진 숨은 의미와 역할을 완벽하게 파헤쳐 보겠습니다.

### 📌 실습 HTML 소스코드 (Live Preview 활용)
> 💡 코드의 변경 사항을 브라우저에서 매번 새로고침하지 않고 VS Code 내부에서 실시간으로 띄워 확인하기 위해 **Live Preview 확장 프로그램**을 설치하여 띄워놓고 직관적으로 매칭하며 실습을 진행했습니다.

```html
<!DOCTYPE html>
<html>
    <!-- 웹 페이지에 대한 설정/정보 -->
    
    <head>
        <title>My Site</title>
    </head>

        <!-- 화면에 그려지는 영역 -->
    <body>

        <!-- heading -->
        <h1> 오즈 뉴스</h1>

        <!-- horizontal line -->
        <hr>

        <h2> [속보] 파이썬 개발자 채용 증가 </h2>

        <!-- 문단 paragraph -->
        <p>
            최근 AI 열풍으로 파이썬 개발자 수요가 늘고 있다
        </p>

    </body>

</html>
